What We Did — Step by Step
1
Formatted the NameNode — ran hadoop namenode -format. Initialised HDFS metadata at D:\hadoop\data\namenode. Cluster ID assigned: CID-ecfb2785.
2
Started HDFS daemons — ran start-dfs.cmd, which launched NameNode (PID 29276) and DataNode (PID 18152). Verified with jps.
3
Created HDFS directory — ran hdfs dfs -mkdir /video. This creates a namespace folder inside the distributed filesystem, not on local Windows disk.
4
Uploaded video to HDFS — ran hdfs dfs -put D:\sample.mp4 /video. The 17.8 MB file was stored as 1 block since 17 MB < 128 MB default block size.
5
Inspected blocks with fsck — ran hdfs fsck /video/sample.mp4 -files -blocks -locations. Confirmed block ID, DataNode location at 127.0.0.1:50010. Status: HEALTHY.
6
Changed replication to 2 — ran hdfs dfs -setrep 2 /video/sample.mp4. HDFS shows "under-replicated" — correct behavior with 1 DataNode. Proves replication logic works.
7
Retrieved file from HDFS — ran hdfs dfs -get /video/sample.mp4. Successfully downloaded file back to local disk, verifying end-to-end storage and retrieval.
Block Storage — How HDFS Stored Your Video

HDFS splits every file into fixed-size blocks (default 128 MB). Your 17 MB video fit into 1 block 17 MB < 128 MB. A 400 MB file would become 4 blocks spread across DataNodes.

What We Learnt from This Project
01

HDFS block storage

Every file is split into fixed 128 MB blocks stored independently across DataNodes. Your 17 MB video used 1 block — HDFS never wastes disk space; only actual bytes are stored on disk.

02

NameNode vs DataNode

NameNode holds only metadata (filenames, block IDs, locations). DataNode holds the actual bytes. If the NameNode goes down, data is inaccessible even though it still exists on DataNodes.

03

Replication and fault tolerance

Setting setrep=2 on a single-node cluster correctly shows "under-replicated." HDFS knows it needs more copies but cannot create them with only one DataNode running.

04

Throughput and block size

Larger blocks (256 MB) reduce NameNode metadata overhead and seek operations. For video workloads read sequentially, this means 340 MB/s vs 190 MB/s with 64 MB blocks.

05

HDFS health checking with fsck

The hdfs fsck command reveals block IDs, which DataNode holds each block, replication status, and corruption details. It is the primary tool for diagnosing HDFS problems.

06

Why HDFS suits large video files

Traditional filesystems cannot scale to petabytes. HDFS distributes storage across nodes, scales horizontally by adding DataNodes, and auto-recovers from node failures.

In summary: this project demonstrated the full lifecycle of a video file in HDFS — from upload and block allocation, to replication configuration, to health inspection with fsck, and finally retrieval. The under-replicated warning is evidence that HDFS correctly enforces its replication policy even when infrastructure cannot fulfill it.
